# Mnemosyne v0.1 — Codex Onboarding and Modes

**Status:** draft (2026-05-28); pending owner review and refinement
**Supersedes (once locked):** the 4-modes treatment in `v0.1-design.md` §5.1 and §5.1.1–5.1.4. Crypto-mechanics from §3 (wrap-CK pattern), §3.4 (3-layer storage), §5.3 (herotag), and §5.7 (mnemosyne.CODEX) remain canonical and are referenced here, not duplicated.
**Authors:** Mihai (owner) + Claude (Opus session, 2026-05-28)

This document specifies how a user onboards onto Mnemosyne, what every codex looks like across the four identity modes, and how the wizard and preset profiles work for each onboarding path. It folds in all refinements from the 2026-05-28 design discussions: the Standard+Smart double-Apollo CodexID, the three wizard primitives, the CodexGuard / Duo Pure Prime entities, the off-chain Schnorr cosign model, and the preset profiles.

---

## 1. Glossary — every entity in a Mnemosyne codex

| Entity | Type | Generated how | Role |
|---|---|---|---|
| **Codex Identity (CodexID)** | Apollo Standard + Smart double pubkey | Derived from a 2048-bit seed split in half (each half → one Apollo keypair) | Public ID for the codex, format `₱.STANDARD.N.SMART` |
| **CodexID Standard private key** | Apollo private key | Derived from Standard half of seed | Off-chain daily auth + the key whose derivation wraps CK |
| **CodexID Smart private key** | Apollo private key | Derived from Smart half of seed | Off-chain cosign for sensitive operations (Guardian role) |
| **Codex Key (CK)** | Random 256-bit AES key | Random at signup; never changes for the codex's life | Encrypts the entire codex blob |
| **wrap_seed** | AES-256-GCM ciphertext of CK | Wrapped under key derived from CodexID Standard private key | Undeletable universal recovery wrap — present in every codex regardless of mode |
| **Prime Codex Seed** (custom-Primitive-3 path) | IKadenaSeed | User-provided Kadena Koala 24-word seed | Source of CodexGuard + Duo Pure Prime via positions #0/#1/#2; the seed itself is flagged undeletable |
| **CodexGuard** | IPureKeypair (Kadena ED25519) | Pure-mode autogenerated OR derived from Prime Codex Seed position #0 | Signs every Pact tx against `mnemosyne.CODEX` |
| **Duo Pure Prime** | Two IPureKeypair entries (Kadena ED25519) | Pure-mode autogenerated OR derived from Prime Codex Seed positions #1+#2 | Back the CodexPrime Ouronet account (one for payment, one for account guard) |
| **CodexPrime account** | IOuroAccount, `isPrime: true` | Derived from CodexID seed via Dalos curve, OR from a separate Dalos seed | The codex's primary on-chain wallet account |

### Universal invariants (every codex, regardless of mode or onboarding path)

- Exactly ONE CodexID (Standard + Smart pair). IMMUTABLE post-creation.
- Exactly ONE registered entry in `mnemosyne.CODEX.codex-identities` on Stoa chain.
- Exactly ONE CodexGuard. Undeletable.
- Exactly TWO Kadena keys backing the CodexPrime account (either as Duo Pure Prime IPureKeypairs, or as Prime Codex Seed positions #1+#2). Undeletable in either form.
- The codex blob is encrypted with CK. CK is always wrapped by `wrap_seed`. CK is additionally wrapped by zero or more mode-layer wraps (password / email-recovery / SMS-recovery / facts).
- `wrap_seed` is mode-independent and undeletable — every codex has it, the seed is the universal recovery factor.

## 2. The three wizard primitives

Every codex creation walks the user through three independent primitive choices. Each primitive has an **autopilot** variant (system makes the choice) and a **custom** variant (user provides input).

### 2.1 Primitive 1 — Codex Identity seed

Generates the 2048 bits that become the CodexID (Apollo Standard + Smart double keypair).

**Autopilot variants** — user picks one of:

| Variant | Word count | Source | Approx entropy |
|---|---|---|---|
| Chainweaver | 12 | BIP39 English | ~132 bits |
| **Koala (default)** | **24** | **BIP39 English** | **~264 bits** |
| 2× Koala | 48 | BIP39 English | ~528 bits |
| 3× Koala | 72 | BIP39 English | ~792 bits |
| … | … | … | … |
| 10× Koala | 240 | BIP39 English | ~2640 bits |
| ☑ **Batshit Crazy** mode | up to 240 | Dictionary words REPLACED by random-length sequences of random Dalos glyphs | depends on length |
| ☑ **Ludicrous** mode | 256 words × 256 glyphs each | All Dalos glyphs, full-length | ~524k bits (cryptographically infinite) |

Beyond ~1024 bits the extra entropy is mathematically wasted (Apollo curve operates at 1024-bit-per-half precision), but higher-entropy modes are offered for users who want the visual / psychological assurance.

**Custom variant** — user provides 2048 bits of entropy in any of these equivalent formats:

- Word sequence (Dalos glyphs, 1–512 words, any length per word)
- 2048-bit bitstring
- Bitmap (e.g., 32×64 or 45×46 grid — paste or scan)
- Base-10 scalar (~616-digit number)
- Base-49 scalar (Apollo curve's natural base)

The SPA accepts any format and converts to canonical 2048-bit bytes before derivation.

**Recovery-via-derivation property**: when the user enters a seed at the Codex Identity step, the wizard derives the Standard Apollo pubkey and checks it against the Stoa chain (`mnemosyne.CODEX.codex-identities`). If the address is already registered, the wizard branches to **recovery flow** automatically — the user is recovering an existing codex, not creating a new one. This is the universal recovery mechanism: knowing the seed = owning the codex.

### 2.2 Primitive 2 — CodexPrime account seed

Generates the CodexPrime Ouronet account (the codex's primary on-chain wallet).

**Autopilot variant**: reuse Primitive 1's seed via the Dalos curve to derive the CodexPrime account. Zero extra friction — the same seed serves both Apollo (for CodexID) and Dalos (for CodexPrime).

**Custom variants**:
- Provide a separate Dalos-curve seed
- Provide a legacy Kadena Koala 24-word seed (compat path for users coming from OuronetUI; in this case the same seed may also serve Primitive 3)

### 2.3 Primitive 3 — CodexGuard + Duo Pure Prime

Generates the three Kadena keys: 1 CodexGuard + 2 Duo Pure Prime.

**Autopilot variant**: generate all 3 keys directly as pure-mode IPureKeypair entries. Each is flagged undeletable. No mnemonic backs them — the keys ARE the entities, stored encrypted inside the codex blob.

**Custom variant**: user provides a Kadena Koala 24-word seed. The seed becomes the **Prime Codex Seed** with `isPrime: true` and `isUnremovable` on positions #0/#1/#2:
- Position #0 → CodexGuard
- Position #1 → CodexPrime payment key
- Position #2 → CodexPrime account guard key

**Mixed variant** (Advanced wizard tab): CodexGuard pure-mode + Duo Pure Prime from a seed, or vice versa. Allowed but hidden behind the Advanced tab so default UX stays simple.

## 3. The Codex Identity double-Apollo construction

### 3.1 Standard + Smart halves

The 2048-bit seed material splits in half. Each half is run through the Apollo curve independently to produce one Apollo keypair. The two public keys combine into the canonical CodexID address format:

```
₱.<Standard-encoded-pubkey>.N.<Smart-encoded-pubkey>
```

Display, share, store this full string. It's the codex's unique identifier and serves as the `mnemosyne.CODEX.codex-identities` row key (or split into two pubkey columns server-side — see §10 open items).

### 3.2 Why two halves — the cosign / Guardian model

The two private keys serve different authorization tiers:

- **Standard half** = the "PIN": used daily, kept online, derives the key that wraps CK, authorizes routine operations
- **Smart half** = the "Guardian": kept offline, never wraps CK, required as a cosigner for sensitive operations

Smart **never replaces** Standard (rotation is impossible — both halves derive from the same seed; replacing one means replacing both, which means a new CodexID, which violates immutability). Instead, Smart **cosigns** sensitive operations alongside Standard.

This mirrors the MultiversX X-Portal Guardian feature: a permanent secondary signer that adds an authorization tier for high-stakes transactions, without ever replacing the primary signer.

### 3.3 What requires cosign vs Standard-alone

**Standard alone authorizes (daily operations):**

- Login (unlock CK via wrap_seed)
- View / read the codex
- Add / remove address book entries
- Add / remove watch list entries
- Add / remove regular Kadena seeds and Ouronet accounts
- Change Mnemosyne password (Mode 1 layer)
- Update arweave-tx-id pointer on chain (frequent backup operation)
- Generate / delete regular Pure Key Pairs (the non-Prime, non-Guard ones)

**Standard + Smart cosign required (sensitive operations):**

- Add / remove Guardians (changes recovery surface)
- Rotate the CodexGuard key
- Rotate the Duo Pure Prime keys
- Convert a Pure Key Pair to / from Duo Pure Prime status
- Add / remove identity modes (any Mode-layer transition, e.g., Mode 4 → Mode 1)
- Claim or release a herotag
- Trigger codex-level destructive operations (if any are ever supported)

### 3.4 Pact verification constraint — and how we route around it

Apollo-curve Schnorr signatures **cannot be verified inside Pact**. Pact's native sig verification is Kadena ED25519. Implementing Apollo verification in pure Pact code would be either gas-prohibitive or infeasible. Modifying the Pact language to add Apollo support is non-trivial and out of scope for v0.1.

The design routes around this cleanly:

- **All on-chain authorization uses CodexGuard** (Kadena key, Pact-native verifier).
- **All off-chain authorization uses CodexID Standard / Smart Apollo signatures**, verified by the SPA and the Mnemosyne backend before any chain tx is constructed.

For sensitive operations:

```
1. User initiates the sensitive operation (e.g., "Add Guardian for account 0xABC")
2. SPA constructs the operation payload (canonical bytes)
3. SPA requires Standard Apollo signature → verifies against chain-stored Standard pubkey
4. SPA requires Smart Apollo signature → verifies against chain-stored Smart pubkey
5. If both verify → SPA constructs the Pact tx to mnemosyne.CODEX
6. SPA signs the Pact tx with CodexGuard (Kadena, single-sig at chain level)
7. SPA submits to Mnemosyne backend with all three signatures attached
8. Backend re-verifies all three signatures before persisting any state change
9. Backend forwards Pact tx to Stoa chain
10. Stoa verifies only the Kadena CodexGuard signature (its native primitive)
11. Tx confirmed; backend state updated
```

The dual-Apollo cosign is enforced TWICE off-chain (SPA + backend). The chain doesn't know about Apollo at all — it just sees CodexGuard signing.

### 3.5 Off-chain Schnorr verification — concrete use cases

The chain stores the two Apollo pubkeys publicly. Consumer apps and the Mnemosyne backend can verify Apollo signatures off-chain at any time.

**Standard-only verification (low-friction "this is me" proof):**

- **Cross-domain codex linking.** User has a codex on Mnemosyne. Visits a new consumer (e.g., OuronetUI at `wallet.ouro.network` after the hard-cutoff migration). Consumer asks for CodexID → issues random challenge → user signs from Mnemosyne SPA with Standard → consumer verifies against on-chain Standard pubkey → accepts CodexID for local use. No re-onboarding needed.
- **API session tokens** issued by consumers, bound to Standard pubkey.
- **Identity assertions in herotag-based messaging / DMs**: prove you really are the holder of `@mihai`.
- **Sign-in-with-CodexID** on third-party services (OAuth-pattern over Apollo).
- **Browser extension auth**: extension verifies it's running for the codex owner before exposing codex data.
- **Signed content sharing**: "share this address book entry" attestations signed by Standard.

**Standard + Smart cosign verification (high-stakes "yes I really mean it"):**

- **Cross-domain linking to high-stakes consumers** (AH hub admin, financial integrations).
- **Approving a Guardian designation request from another device**.
- **Authorizing a consumer to make sensitive operations** in your codex on your behalf.
- **Tier-changing operations on Mnemosyne** itself (Mode 4 → Mode 1: removing safety nets).
- **Full codex export** including all private key material.
- **Codex migration to a fresh CodexID** (irreversible).

The pattern: Standard suffices for "this is me"; Standard + Smart together signal "this is me AND I really mean it."

### 3.6 Asymmetric splits and the Smart-bigger rule

For odd-word-count seeds, the **Smart half always gets the bigger share**. For example:

- 7 words → 3 Standard + 4 Smart
- 17 words → 8 Standard + 9 Smart
- 251 words → 125 Standard + 126 Smart

Users may override via the wizard's Advanced tab (e.g., 17 words with custom 11/6 split). The entropy contribution of split-position variability is negligible (~log₂ of split-position count) against the seed's overall entropy, so this is a configurability feature, not a security feature.

### 3.7 Maximum entropy ceiling

- Per-half maximum: **256 words** (dictionary words OR Ludicrous-mode 256-glyph blobs)
- Total maximum: **512 words**

Beyond this, additional bits don't change the cryptographic story (the Apollo curve already operates at 1024-bit-per-half precision).

## 4. The three onboarding paths

Orthogonal to mode choice. Any of the 4 modes can be reached via any of the 3 paths.

### 4.1 Path A — New codex (no prior codex anywhere)

User has nothing. Wizard creates everything from scratch.

**Wizard steps:**
1. Mode selection (1/2/3/4)
2. Primitive 1 (CodexID seed) — autopilot or custom
3. Primitive 2 (CodexPrime account seed) — autopilot (reuse Primitive 1) or custom
4. Primitive 3 (CodexGuard + Duo Pure Prime) — autopilot or custom
5. Mode-layer questions (per mode; see §5)
6. Chain registration via `mnemosyne.CODEX.register-codex-identity`
7. Codex blob assembled, encrypted, uploaded to Mnemosyne backend (L1 storage per v0.1-design.md §3.4)
8. Display the user's CodexID address + Smart half + (if Mode 1/2 with self-managed seed) save-your-seed prompt
9. Done

### 4.2 Path B — Import from browser localStorage

User has an existing OuronetUI codex sitting in their browser at `devwallet.ouronetwork.io` (or wherever). Wizard imports it into Mnemosyne.

**Wizard steps:**
1. Read existing codex from browser localStorage via `LocalStorageCodexAdapter`
2. Prompt for existing OuronetUI codex password → unlock → get existing snapshot
3. Mode selection (1/2/3/4)
4. Primitive 1 (CodexID seed) — autopilot (derive from existing Prime Codex Seed via Apollo curve, then check chain for collision/recovery branch) or custom (new seed)
5. Primitive 2 — **SKIPPED** (existing codex already has a CodexPrime; wizard uses it)
6. Primitive 3 — autopilot adds a fresh CodexGuard in pure mode; existing Kadena seed positions #1+#2 continue to back the CodexPrime (flagged unremovable now if not already, with `isPrime: true` upgrade applied)
7. Mode-layer questions
8. Chain registration of the (possibly newly-derived) CodexID
9. Re-encrypt snapshot under a fresh CK; upload to L1
10. Prompt user to wipe browser localStorage (optional, with explicit confirm)
11. Done

### 4.3 Path C — Import from JSON backup file

User has a JSON file (downloaded OuronetUI backup, Mnemosyne export, or any compatible third-party tool's export). Wizard imports it.

**Wizard steps:** identical to Path B except step 1 reads from the uploaded file instead of localStorage.

## 5. The four identity modes

Modes are strictly cumulative: Mode N includes everything from Mode N-1 plus additional credentials, wraps, and/or login methods.

### 5.1 Mode 1 — Privacy

The base mode. Zero PII collected. Two unwrap paths (universal seed + password convenience).

**What the user provides at signup:**
- Username (lookup identifier on Mnemosyne, non-PII, must be unique server-side)
- Password
- Plus the wizard's 3 primitives

**Wraps of CK in the codex:**
- `wrap_seed` (universal, derived from CodexID Standard private key — undeletable)
- `wrap_password` (derived from password — Mode 1 layer)

**Server stores per Mode 1 user:**
- `username` (PK)
- `apollo_standard_pubkey`, `apollo_smart_pubkey` (the two halves of CodexID)
- `password_verifier` (Argon2id hash of password + salt)
- `salt_pw`, `salt_verifier`
- The codex blob (single concatenated format per v0.1-design.md §3.4)
- Timestamps

**Login flow:**
1. User enters username + password
2. Server returns codex blob + salts + password_verifier check passes
3. SPA derives password key (Argon2id with salt_pw) → unwraps CK from wrap_password
4. SPA decrypts codex → user is in

**Recovery flow (forgot password):**
1. User goes to recovery URL
2. Enters Codex Identity seed (any of 4 input formats)
3. SPA derives Apollo Standard private key → wizard checks chain for the CodexID
4. If found → SPA fetches codex blob from server (read-only, no password) → unwraps CK from wrap_seed → decrypts codex
5. SPA prompts for new password → re-wraps CK with new password key → uploads new wrap_password to server
6. User is in

**Custody property:** STRICT self-custody. Mnemosyne cannot decrypt anything with what it holds. Lose both seed AND password → codex permanently inaccessible.

### 5.2 Mode 2 — Wallet (= Mode 1 + wallet-signature login)

Adds an alternative login method via Schnorr signature against the CodexPrime account. No new wrap of CK — purely a server-side auth surface addition.

**What the user provides at signup:** everything from Mode 1, plus implicit "use CodexPrime as the wallet-auth account" (no choice — always the codex's prime account, per locked design).

**Wraps of CK in the codex:** same as Mode 1 (no additions). Mode 2 does NOT add a wrap.

**Server stores (additions to Mode 1):**
- `codex_prime_address` (Ѻ.xxx)
- `codex_prime_pubkey` (Dalos-curve pubkey, for Schnorr challenge verification)

**Login flow (Mode 2-specific path):**
1. User enters CodexPrime address (or pastes from QR / browser extension memory)
2. Server returns a random challenge nonce (32 bytes, valid 60s) + codex blob
3. SPA prompts user to unlock (via locally-cached codex password OR re-typed CodexPrime mnemonic) → derives CodexPrime private key
4. SPA Schnorr-signs the challenge → server verifies signature against stored `codex_prime_pubkey` → session token issued
5. SPA derives CodexID Standard private key from the same root seed → unwraps CK from wrap_seed → decrypts codex
6. User is in

**Login flow (Mode 1-style still works):** username + password, exactly as Mode 1.

**Recovery flow:** same as Mode 1 (universal seed-derivation path).

**Custody property:** STRICT self-custody.

### 5.3 Mode 3 — Convenience (= Mode 2 + email reset path)

Adds email-based password reset.

**What the user provides at signup:** everything from Mode 2, plus email address.

**Wraps of CK in the codex (additions to Mode 2):**
- `wrap_email_recovery` (CK wrapped with key derived from Email Recovery Secret)

**The Email Recovery Secret:**
- Random 256-bit value generated at signup
- Emailed ONCE to the user immediately, with strong "save this — we cannot resend" framing
- Mnemosyne stores ONLY the wrap (not the secret)
- The secret is the user's responsibility from email-send onwards

**Server stores (additions to Mode 2):**
- `email` (PK lookup for password reset flow; otherwise auth-irrelevant)
- `email_verified_at`

(The `wrap_email_recovery` lives in the codex blob, not as a separate server column.)

**Login flow:** same as Mode 2 (username+password and/or wallet-signature). Email is for recovery, not login.

**Recovery flow (in addition to Mode 1's seed-recovery):**
1. User clicks "Forgot password" → enters email
2. Mnemosyne emails a magic link (1h validity)
3. User clicks link → SPA loads → server validates token → SPA prompts "paste your Email Recovery Secret"
4. User pastes secret → SPA derives email_recovery key → unwraps CK from wrap_email_recovery
5. SPA prompts for new password → re-wraps CK with new password key → uploads new wrap_password
6. User is in

**Custody property:** hybrid. Recovery surfaces include the user-held seed AND the user-held Email Recovery Secret. Neither involves operator-decryptable material.

### 5.4 Mode 4 — Streamlined (= Mode 3 + phone + facts + SMS recovery)

Adds phone-based recovery and personal-facts recovery for grandma-tier accessibility.

**What the user provides at signup:** everything from Mode 3, plus:
- Phone number (with SMS verification at signup)
- Personal facts (4–7 Q&A: mother's maiden name, first pet, etc.)
- A chosen "secret word" (pepper) — only the user knows this

**Wraps of CK in the codex (additions to Mode 3):**
- `wrap_sms_recovery` (CK wrapped with key derived from SMS Recovery Secret)
- `wrap_facts` (CK wrapped with Argon2id-128M-derived key from canonical facts + secret word)

**Server stores (additions to Mode 3):**
- `phone` (E.164 format, PK lookup for SMS recovery flow)
- `facts_verifier` (Argon2id hash of canonical facts, for "did you get them right?" check)
- `salt_facts`, `salt_facts_hash`
- `phone_verified_at`

**Login flow:** same as Mode 3 (username+password and/or wallet-signature). Phone is for recovery, not login (per locked SMS-only-auth rejection from v0.1-design.md §5.2).

**Recovery flow (in addition to Mode 1/2/3 paths):**
1. User clicks "Forgot password" → enters phone
2. Mnemosyne sends SMS code → user enters → server validates
3. SPA shows two paths:
   - **Path A (best):** paste SMS Recovery Secret from signup SMS → derive sms_recovery key → unwrap CK
   - **Path B (backup):** answer personal facts + secret word → SPA computes canonical facts → checks against `facts_verifier` → on match, derive facts key → unwrap CK
4. SPA prompts for new password → re-wraps CK with new password key → uploads new wrap_password
5. User is in

**Custody property:** hybrid, grandma-leaning. `wrap_facts` is the only wrap theoretically vulnerable to operator brute-force IF facts are weak and the operator has full DB access — Argon2id-128M makes this slow and expensive but not cryptographically impossible. Documented as the explicit trade-off for grandma accessibility.

## 6. Mode is strictly cumulative

Each mode includes ALL of the surface of lower modes:

| Mode | Identifier(s) | Credentials | Login methods | Recovery surfaces (additive) |
|---|---|---|---|---|
| 1 | username | password | username+password | seed |
| 2 | username, CodexPrime addr | password | username+password, wallet-sig | seed |
| 3 | username, CodexPrime addr, email | password | username+password, wallet-sig | seed, email-secret |
| 4 | username, CodexPrime addr, email, phone | password, facts | username+password, wallet-sig | seed, email-secret, SMS-secret, facts |

### Mode-switching

Users can change modes after signup.

**Adding a mode (e.g., Mode 1 → Mode 3):**
1. User logged in (SPA has CK in memory)
2. Wizard collects the new mode's credentials (email + Email Recovery Secret generation, etc.)
3. SPA computes the new wrap(s) of CK
4. SPA uploads new wraps + new server-side row data
5. Now user can use both Mode 1 (username+password) AND Mode 3 (email reset path)

**Removing a mode (e.g., Mode 3 → Mode 2):**
1. User logged in
2. Wizard confirms: "this deletes the email-reset path; you'll only be able to recover via your seed"
3. **24-hour cooling-off period** begins (mitigates account takeover via single compromised mode)
4. After cooling-off + explicit confirm, server deletes the higher-mode wraps + metadata
5. Codex blob and CK untouched; lower-mode wraps untouched

**Invariants preserved across all mode transitions:**
- CK never changes
- Codex blob never re-encrypts (only the wrap set evolves)
- CodexID never changes
- CodexGuard never changes
- Duo Pure Prime never changes
- The on-chain `mnemosyne.CODEX` row never changes (mode-layer is purely off-chain)

## 7. Preset profiles

Pre-configured wizard sequences that minimize friction for specific user populations.

### 7.1 Migration preset (silent OuronetUI migration)

For existing OuronetUI users moving to Mnemosyne via the hard-cutoff migration. Designed to require zero user decisions beyond clicking "Migrate."

**Configuration:**
- **Onboarding path:** B (browser localStorage) or C (JSON backup), whichever the user has
- **Primitive 1:** autopilot → derive CodexID from existing Prime Codex Seed via Apollo curve
- **Primitive 2:** existing (no action — codex already has a CodexPrime)
- **Primitive 3:** autopilot → CodexGuard auto-generated in pure mode; existing Kadena seed positions continue to back CodexPrime
- **Mode:** Mode 2 (Wallet) — user identifies with their existing wallet; no new password to invent

**User experience:** click "Migrate to Mnemosyne" on devwallet.ouronetwork.io → unlock codex with existing password → confirm. Done.

### 7.2 Movie Platform preset (frictionless grandma onboarding)

For users signing up on the movie platform who don't know what a codex is.

**Configuration:**
- **Onboarding path:** A (new codex from scratch)
- **Primitive 1:** autopilot → 24 Koala words, generated silently, NEVER shown to user
- **Primitive 2:** autopilot → reuse Primitive 1's seed via Dalos curve
- **Primitive 3:** autopilot → CodexGuard + Duo Pure Prime, all pure-mode
- **Mode:** Mode 4 (Streamlined) — phone + password + facts

**User experience:** "Sign up for movies" → phone + verify code → password → answer 4 personal facts + chosen secret word → done. Codex exists in the background; user has no awareness of it.

**Graduation path:** if the user later wants to graduate to a self-custody mode (Mode 1 or Mode 2), the Mnemosyne SPA reveals the previously-hidden material (CodexID seed words, Smart half, Duo Pure Prime keys, CodexGuard) with strong "save these now, you'll need them" framing.

### 7.3 Optional minimal-friction variant of the Movie Platform preset

For users who want one touch of agency but otherwise full autopilot: offer a single optional step at signup — "Do you want to define your own Codex Identity seed words?" — defaulting to "no, just create one for me." If yes, the user types their own seed; everything else stays autopilot.

## 8. The full onboarding matrix

Every codex creation in Mnemosyne resolves to a combination of:

- Onboarding path (A / B / C)
- Mode (1 / 2 / 3 / 4)
- Per-primitive autopilot/custom choices

The wizard exposes every combination; preset profiles are pre-selected templates. A power user can deviate at any step.

| Path × Mode | What runs |
|---|---|
| A1, A2, A3, A4 | Full wizard: 3 primitives + Mode N layer |
| B1, B2, B3, B4 | Import wizard: Primitive 1 + Primitive 3 only (P2 skipped, existing CodexPrime used) + Mode N layer |
| C1, C2, C3, C4 | Same as B but from JSON file |

**12 combinations total.** Migration preset = B2 with all autopilots. Movie Platform preset = A4 with all autopilots.

## 9. Recovery flows summary

All modes share the **universal recovery path**: provide the Codex Identity seed (in any of 4 input formats) → derive Standard private key → unwrap_seed → reset whatever needs resetting. This works even without Mnemosyne backend access if the user has the codex blob (from L1, L2 Arweave, or a JSON backup).

Mode-specific recovery surfaces ADD to this universal path:

| Mode | Universal | Mode-specific |
|---|---|---|
| 1 | seed | (none additional) |
| 2 | seed | (none — wallet-sig is for LOGIN, not RECOVERY) |
| 3 | seed | email magic link + Email Recovery Secret |
| 4 | seed | SMS code + (SMS Recovery Secret OR personal facts + secret word) |

**Disaster recovery (Mnemosyne entirely gone):**
- If user has Codex Identity seed AND a copy of the encrypted blob (from Arweave L2 backup, JSON export, or any third party) → full recovery via standalone tool / any Mnemosyne-compatible client
- If user has only their seed but no blob copy → must rebuild codex from on-chain accounts (whatever they registered on Stoa is still there; address book + watchlist + Mode-specific settings lost)
- If user has only the blob but no seed → blob is opaque; no recovery
- If both lost → no recovery (same outcome as today's "lose seed = lose codex")
