# Mnemosyne v0.1 — Codex Onboarding and Modes

**Status:** revised (2026-05-30) — incorporates the locked login-model decisions D1–D4 from [`v0.1-open-decisions.md`](./v0.1-open-decisions.md). Pending owner re-read.
**Supersedes (once locked):** the 4-modes treatment in `v0.1-design.md` §5.1 and §5.1.1–5.1.4. Crypto-mechanics from §3 (wrap-CK pattern), §3.4 (3-layer storage), §5.3 (herotag), and §5.7 (mnemosyne.CODEX) remain canonical and are referenced here, not duplicated.
**Authors:** Mihai (owner) + Claude (Opus sessions, 2026-05-28 / 2026-05-30)

This document specifies how a user onboards onto Mnemosyne, what every codex looks like across the identity modes, and how the wizard and preset profiles work for each onboarding path.

> **2026-05-30 revision — what changed (see decisions D1–D4):**
> - **Mode 1 is no longer username + password.** It is **key-possession login**: a per-device **passkey** for daily use, the **Standard key** to bootstrap a device, **either key** to recover. Zero server-side password door. A browser "secure keyboard" against a compromised host is impossible and out of scope (D1).
> - **The Smart key is now a second unlock/recovery root**, and **adding a device requires both keys** (Standard + Smart, 2-of-2). Unlock needs only one factor; expanding the trust surface needs both (D1).
> - **Username + password is demoted to a Mode 2 add-on** — a web2 familiarity bridge, not the base. The ladder is now **1 Sovereign / 2 +Password / 3 +Email / 4 +Phone(deferred to v0.2)** (D2).
> - **CodexID uses two independent seeds** (Seed-A Standard, Seed-B Smart), not one split phrase, for a clearer user mental model (D3).
> - **Mode 4 (Phone/SMS) is deferred to v0.2.** v0.1 ships Modes 1–3 (D2 + A4). A cold-signer / mobile-app tier is future R&D (D4).

## 1. Glossary — every entity in a Mnemosyne codex

| Entity | Type | Generated how | Role |
|---|---|---|---|
| **Codex Identity (CodexID)** | Apollo Standard + Smart double pubkey | Standard from **Seed-A**, Smart from **Seed-B** (two independent seeds, per D3) | Public ID for the codex, format `₱.STANDARD.N.SMART` |
| **CodexID Standard private key** | Apollo private key | Derived from **Seed-A** | The "this is me" key: bootstraps a device, unlocks CK via `wrap_seed`, signs daily off-chain ops. Treat as the burnable daily key. |
| **CodexID Smart private key** | Apollo private key | Derived from **Seed-B** | The guardian "crown jewel": second unlock/recovery root, required (with Standard) to enroll a device or cosign sensitive ops. Kept cold. |
| **Codex Key (CK)** | Random 256-bit AES key | Random at signup; never changes for the codex's life | Encrypts the entire codex blob |
| **wrap_seed** | AES-256-GCM ciphertext of CK | Wrapped under key derived from CodexID **Standard** private key (Seed-A) | Undeletable recovery wrap — present in every codex. Unwrapped by typing the Standard key. |
| **wrap_smart** | AES-256-GCM ciphertext of CK | Wrapped under key derived from CodexID **Smart** private key (Seed-B) | Second undeletable recovery root (per D1). Lets *either* seed recover the codex — removes the single-point-of-loss on Seed-A. |
| **wrap_passkey[]** | Array of AES-256-GCM ciphertexts of CK | Each entry wrapped under a key from one enrolled device's passkey (WebAuthn PRF) | The daily door(s). One entry per enrolled device; revocable per device. Strong, hardware-bound, phishing-resistant. |
| **Prime Codex Seed** (custom-Primitive-3 path) | IKadenaSeed | User-provided Kadena Koala 24-word seed | Source of CodexGuard + Duo Pure Prime via positions #0/#1/#2; the seed itself is flagged undeletable |
| **CodexGuard** | IPureKeypair (Kadena ED25519) | Pure-mode autogenerated OR derived from Prime Codex Seed position #0 | Signs every Pact tx against `mnemosyne.CODEX` |
| **Duo Pure Prime** | Two IPureKeypair entries (Kadena ED25519) | Pure-mode autogenerated OR derived from Prime Codex Seed positions #1+#2 | Back the CodexPrime Ouronet account (one for payment, one for account guard) |
| **CodexPrime account** | IOuroAccount, `isPrime: true` | Derived from CodexID seed via Dalos curve, OR from a separate Dalos seed | The codex's primary on-chain wallet account |

### Universal invariants (every codex, regardless of mode or onboarding path)

- Exactly ONE CodexID (Standard + Smart pair). IMMUTABLE post-creation.
- Exactly ONE registered entry in `mnemosyne.CODEX.codex-identities` on Stoa chain.
- Exactly ONE CodexGuard. Undeletable.
- Exactly TWO Kadena keys backing the CodexPrime account (either as Duo Pure Prime IPureKeypairs, or as Prime Codex Seed positions #1+#2). Undeletable in either form.
- The codex blob is encrypted with CK. CK is always reachable via `wrap_seed` (Standard) **and** `wrap_smart` (Smart) — the two undeletable recovery roots. CK is additionally reachable via zero or more `wrap_passkey[]` entries (one per enrolled device) and zero or more mode-layer wraps (Mode 2 password / Mode 3 email-recovery / Mode 4 SMS-recovery + facts).
- `wrap_seed` and `wrap_smart` are mode-independent and undeletable — every codex has both; either seed recovers the codex.
- **Tiered authority (per D1):** *unlocking* CK needs **one** factor (any passkey, or Standard, or Smart). *Enrolling a new device* or any *sensitive op* needs **both** keys (Standard + Smart, 2-of-2). This limits a single stolen key to in-session access — it cannot plant a standing backdoor device.

## 2. The three wizard primitives

Every codex creation walks the user through three independent primitive choices. Each primitive has an **autopilot** variant (system makes the choice) and a **custom** variant (user provides input).

### 2.1 Primitive 1 — Codex Identity seed(s)

Generates the entropy that becomes the CodexID (Apollo Standard + Smart double keypair).

**Default (per D3): two independent seeds.** The user composes **Seed-A → Standard** (the daily "this is me" key) and **Seed-B → Smart** (the cold guardian key) as two separate phrases. This is deliberate: two distinct phrases with two distinct roles is a clearer mental model than one phrase the system secretly splits, and it lets the two be stored in different places (breach the convenient one → daily access only, not guardian-gated ops). Each seed is drawn from the same menu below, independently.

**Autopilot / grandma-preset alternative: one split seed.** Where the seed is never shown to the user (Movie Platform preset, §7.2), a single phrase is generated and split (smaller half → Standard, bigger → Smart) — the split is invisible, so the "one artifact" story is simpler. Same crypto envelope either way (D3).

**Autopilot variants** (per seed) — user picks one of:

| Variant | Word count | Source | Approx entropy |
|---|---|---|---|
| Chainweaver | 12 | BIP39 English | ~132 bits |
| **Koala (default)** | **24** | **BIP39 English** | **~264 bits** |
| 2× Koala | 48 | BIP39 English | ~528 bits |
| 3× Koala | 72 | BIP39 English | ~792 bits |
| … | … | … | … |
| 10× Koala | 240 | BIP39 English | ~2640 bits |
| ☑ **Batshit Crazy** mode | up to 240 | Dictionary words REPLACED by random-length sequences of random Dalos glyphs | depends on length |
| ☑ **Ludicrous** mode | 256 words × 256 glyphs each | All Dalos glyphs, full-length | ~524k bits (cryptographically infinite) |

Beyond ~1024 bits the extra entropy is mathematically wasted (Apollo curve operates at 1024-bit-per-half precision), but higher-entropy modes are offered for users who want the visual / psychological assurance.

**Custom variant** — user provides 2048 bits of entropy in any of these equivalent formats:

- Word sequence (Dalos glyphs, 1–512 words, any length per word)
- 2048-bit bitstring
- Bitmap (e.g., 32×64 or 45×46 grid — paste or scan)
- Base-10 scalar (~616-digit number)
- Base-49 scalar (Apollo curve's natural base)

The SPA accepts any format and converts to canonical 2048-bit bytes before derivation.

**Recovery-via-derivation property**: when the user enters a seed at the Codex Identity step, the wizard derives its Apollo pubkey and checks both the Standard and Smart columns against the Stoa chain (`mnemosyne.CODEX.codex-identities`). If it matches a registered codex (as either Seed-A/Standard or Seed-B/Smart), the wizard branches to **recovery flow** automatically — the user is recovering an existing codex, not creating a new one. This is the universal recovery mechanism: knowing *either* seed = owning the codex.

### 2.2 Primitive 2 — CodexPrime account seed

Generates the CodexPrime Ouronet account (the codex's primary on-chain wallet).

**Autopilot variant**: reuse Primitive 1's seed via the Dalos curve to derive the CodexPrime account. Zero extra friction — the same seed serves both Apollo (for CodexID) and Dalos (for CodexPrime).

**Custom variants**:
- Provide a separate Dalos-curve seed
- Provide a legacy Kadena Koala 24-word seed (compat path for users coming from OuronetUI; in this case the same seed may also serve Primitive 3)

### 2.3 Primitive 3 — CodexGuard + Duo Pure Prime

Generates the three Kadena keys: 1 CodexGuard + 2 Duo Pure Prime.

**Autopilot variant**: generate all 3 keys directly as pure-mode IPureKeypair entries. Each is flagged undeletable. No mnemonic backs them — the keys ARE the entities, stored encrypted inside the codex blob.

**Custom variant**: user provides a Kadena Koala 24-word seed. The seed becomes the **Prime Codex Seed** with `isPrime: true` and `isUnremovable` on positions #0/#1/#2:
- Position #0 → CodexGuard
- Position #1 → CodexPrime payment key
- Position #2 → CodexPrime account guard key

**Mixed variant** (Advanced wizard tab): CodexGuard pure-mode + Duo Pure Prime from a seed, or vice versa. Allowed but hidden behind the Advanced tab so default UX stays simple.

## 3. The Codex Identity double-Apollo construction

### 3.1 Two seeds, two Apollo keypairs

Per D3, the CodexID is built from **two independent seeds**:

- **Seed-A** → Apollo curve → the **Standard** keypair (the daily "this is me" key)
- **Seed-B** → Apollo curve → the **Smart** keypair (the cold guardian + second recovery root)

Each seed is drawn independently from the Primitive-1 menu (§2.1) and each produces one Apollo keypair. The two public keys combine into the canonical CodexID address format:

```
₱.<Standard-encoded-pubkey>.N.<Smart-encoded-pubkey>
```

Display, share, store this full string. It's the codex's unique identifier and serves as the `mnemosyne.CODEX.codex-identities` row key (or split into two pubkey columns server-side — see §10 open items).

> **One-split alternative (autopilot only).** Where the seed is never shown to the user (Movie Platform preset, §7.2), a single phrase is generated and split internally — smaller half → Standard, bigger half → Smart — so the "one artifact" story stays simple. The §3.6 Smart-bigger rule applies *only* to this path. In the default two-seed flow there is no split: each seed feeds one keypair directly.

### 3.2 Why two keys — the cosign / Guardian model

The two private keys serve different authorization tiers:

- **Standard** = the daily key: used online, derives the key behind `wrap_seed`, authorizes routine off-chain operations, bootstraps a new device. Treat it as the burnable daily key.
- **Smart** = the Guardian: kept cold, a **second unlock/recovery root** (`wrap_smart`), and required *together with Standard* to enroll a new device or cosign sensitive operations.

Smart **never replaces** Standard (rotation is impossible — replacing either key means a new CodexID, which violates immutability). Instead, Smart serves two roles: an independent way back in (recovery), and the mandatory second signature for anything that expands the codex's trust surface.

This mirrors the MultiversX X-Portal Guardian feature: a permanent secondary signer that adds an authorization tier for high-stakes operations, without ever replacing the primary signer. The difference under D1 is that Smart is also a full recovery root on its own — either seed alone recovers the codex, but neither seed alone can enroll a standing new device.

### 3.3 What requires cosign vs Standard-alone

**One factor authorizes (unlock + daily operations)** — any enrolled passkey, OR the Standard key, OR the Smart key:

- Unlock CK (via `wrap_passkey[]`, `wrap_seed`, or `wrap_smart`)
- View / read the codex
- Add / remove address book entries
- Add / remove watch list entries
- Add / remove regular Kadena seeds and Ouronet accounts
- Update arweave-tx-id pointer on chain (frequent backup operation)
- Generate / delete regular Pure Key Pairs (the non-Prime, non-Guard ones)
- Add / change a Mode 2 password layer (it is only a convenience door, never a recovery root)

**Standard + Smart cosign required (sensitive operations — anything that expands the trust surface):**

- **Enroll a new device** (add a `wrap_passkey[]` entry) — a single stolen key cannot plant a standing backdoor device
- Add / remove Guardians (changes recovery surface)
- Rotate the CodexGuard key
- Rotate the Duo Pure Prime keys
- Convert a Pure Key Pair to / from Duo Pure Prime status
- Add / remove identity modes (any Mode-layer transition)
- Claim or release a herotag
- Trigger codex-level destructive operations (if any are ever supported)

> **Honest limit (per D1):** the 2-of-2 cosign protects *persistence* — it stops an attacker with one key from leaving themselves a permanent way back in. It does **not** stop in-session draining: whoever unlocks CK on a compromised host can read and use the codex for that session. The defense is device hygiene + the ability to revoke a device's passkey wrap, not a magic in-browser blackbox (which is impossible — D1).

### 3.4 Pact verification constraint — and how we route around it

Apollo-curve Schnorr signatures **cannot be verified inside Pact**. Pact's native sig verification is Kadena ED25519. Implementing Apollo verification in pure Pact code would be either gas-prohibitive or infeasible. Modifying the Pact language to add Apollo support is non-trivial and out of scope for v0.1.

The design routes around this cleanly:

- **All on-chain authorization uses CodexGuard** (Kadena key, Pact-native verifier).
- **All off-chain authorization uses CodexID Standard / Smart Apollo signatures**, verified by the SPA and the Mnemosyne backend before any chain tx is constructed.

For sensitive operations:

```
1. User initiates the sensitive operation (e.g., "Add Guardian for account 0xABC")
2. SPA constructs the operation payload (canonical bytes)
3. SPA requires Standard Apollo signature → verifies against chain-stored Standard pubkey
4. SPA requires Smart Apollo signature → verifies against chain-stored Smart pubkey
5. If both verify → SPA constructs the Pact tx to mnemosyne.CODEX
6. SPA signs the Pact tx with CodexGuard (Kadena, single-sig at chain level)
7. SPA submits to Mnemosyne backend with all three signatures attached
8. Backend re-verifies all three signatures before persisting any state change
9. Backend forwards Pact tx to Stoa chain
10. Stoa verifies only the Kadena CodexGuard signature (its native primitive)
11. Tx confirmed; backend state updated
```

The dual-Apollo cosign is enforced TWICE off-chain (SPA + backend). The chain doesn't know about Apollo at all — it just sees CodexGuard signing.

### 3.5 Off-chain Schnorr verification — concrete use cases

The chain stores the two Apollo pubkeys publicly. Consumer apps and the Mnemosyne backend can verify Apollo signatures off-chain at any time.

**Standard-only verification (low-friction "this is me" proof):**

- **Cross-domain codex linking.** User has a codex on Mnemosyne. Visits a new consumer (e.g., OuronetUI at `wallet.ouro.network` after the hard-cutoff migration). Consumer asks for CodexID → issues random challenge → user signs from Mnemosyne SPA with Standard → consumer verifies against on-chain Standard pubkey → accepts CodexID for local use. No re-onboarding needed.
- **API session tokens** issued by consumers, bound to Standard pubkey.
- **Identity assertions in herotag-based messaging / DMs**: prove you really are the holder of `@mihai`.
- **Sign-in-with-CodexID** on third-party services (OAuth-pattern over Apollo).
- **Browser extension auth**: extension verifies it's running for the codex owner before exposing codex data.
- **Signed content sharing**: "share this address book entry" attestations signed by Standard.

**Standard + Smart cosign verification (high-stakes "yes I really mean it"):**

- **Cross-domain linking to high-stakes consumers** (AH hub admin, financial integrations).
- **Approving a Guardian designation request from another device**.
- **Authorizing a consumer to make sensitive operations** in your codex on your behalf.
- **Tier-changing operations on Mnemosyne** itself (Mode 4 → Mode 1: removing safety nets).
- **Full codex export** including all private key material.
- **Codex migration to a fresh CodexID** (irreversible).

The pattern: Standard suffices for "this is me"; Standard + Smart together signal "this is me AND I really mean it."

### 3.6 Asymmetric splits and the Smart-bigger rule (one-split path only)

This rule applies **only to the one-split autopilot path** (§3.1, §7.2) where a single phrase is divided internally. In the default two-seed flow it does not apply — Seed-A and Seed-B are independent and each feeds one keypair whole.

When a single phrase is split, the **Smart half always gets the bigger share** of an odd word count:

- 7 words → 3 Standard + 4 Smart
- 17 words → 8 Standard + 9 Smart
- 251 words → 125 Standard + 126 Smart

Users may override via the wizard's Advanced tab. The entropy contribution of split-position variability is negligible (~log₂ of split-position count) against the seed's overall entropy, so this is a configurability feature, not a security feature.

### 3.7 Maximum entropy ceiling

- Per-seed maximum: **256 words** (dictionary words OR Ludicrous-mode 256-glyph blobs)
- Combined maximum across both seeds: **512 words**

Beyond this, additional bits don't change the cryptographic story (each Apollo keypair already operates at 1024-bit precision).

## 4. The three onboarding paths

Orthogonal to mode choice. Any of the 4 modes can be reached via any of the 3 paths.

### 4.1 Path A — New codex (no prior codex anywhere)

User has nothing. Wizard creates everything from scratch.

**Wizard steps:**
1. Mode selection (1/2/3 — **Mode 4 deferred to v0.2**, per D2/A4)
2. Primitive 1 (CodexID seeds) — autopilot or custom; default is two independent seeds (Seed-A → Standard, Seed-B → Smart)
3. Primitive 2 (CodexPrime account seed) — autopilot (reuse Seed-A) or custom
4. Primitive 3 (CodexGuard + Duo Pure Prime) — autopilot or custom
5. Mode-layer questions (per mode; see §5)
6. Chain registration via `mnemosyne.CODEX.register-codex-identity`
7. Codex blob assembled, encrypted, uploaded to Mnemosyne backend (L1 storage per v0.1-design.md §3.4)
8. **Enroll the first device** — prompt to register a passkey on this device (creates the first `wrap_passkey[]` entry, the daily door). This is allowed at creation with the freshly-held Standard+Smart keys (the 2-of-2 enrollment gate is satisfied because the user holds both seeds at signup).
9. Display the CodexID address + save-both-seeds prompt (Seed-A and Seed-B, clearly labelled "daily" vs "guardian/recovery"), unless an autopilot preset is hiding the seeds (§7.2)
10. Done

### 4.2 Path B — Import from browser localStorage

User has an existing OuronetUI codex sitting in their browser at `devwallet.ouronetwork.io` (or wherever). Wizard imports it into Mnemosyne.

**Wizard steps:**
1. Read existing codex from browser localStorage via `LocalStorageCodexAdapter`
2. Prompt for existing OuronetUI codex password → unlock → get existing snapshot
3. Mode selection (1/2/3 — Mode 4 deferred to v0.2)
4. Primitive 1 (CodexID seeds) — autopilot (derive from existing Prime Codex Seed via Apollo curve, then check chain for collision/recovery branch) or custom (new seeds)
5. Primitive 2 — **SKIPPED** (existing codex already has a CodexPrime; wizard uses it)
6. Primitive 3 — autopilot adds a fresh CodexGuard in pure mode; existing Kadena seed positions #1+#2 continue to back the CodexPrime (flagged unremovable now if not already, with `isPrime: true` upgrade applied)
7. Mode-layer questions
8. Chain registration of the (possibly newly-derived) CodexID
9. Re-encrypt snapshot under a fresh CK; upload to L1
10. Prompt user to wipe browser localStorage (optional, with explicit confirm)
11. Done

### 4.3 Path C — Import from JSON backup file

User has a JSON file (downloaded OuronetUI backup, Mnemosyne export, or any compatible third-party tool's export). Wizard imports it.

**Wizard steps:** identical to Path B except step 1 reads from the uploaded file instead of localStorage.

## 5. The identity modes

The ladder runs from maximum sovereignty to maximum convenience. **Mode 1 is the secure base**; every mode above it *adds* a more familiar (and weaker) door without removing the strong ones. Modes are strictly cumulative: Mode N includes everything from Mode N-1 plus additional credentials, wraps, and/or login methods.

| Mode | Name | Adds | Status |
|---|---|---|---|
| **1** | **Sovereign** | key-possession login: passkey (daily) + Standard key (bootstrap) + either seed (recover) | **v0.1** |
| **2** | **+Password** | a username + password convenience door (web2 familiarity bridge) | **v0.1** |
| **3** | **+Email** | email-based recovery path | **v0.1** |
| 4 | +Phone | phone/SMS recovery + personal-facts recovery (grandma tier) | **deferred to v0.2** |

> The old design had this ladder **inverted** — username + password was Mode 1 (the base everyone got) and crypto-native login was bolted on later. D2 flips it: the strong door is the floor, the weak doors are opt-in additions on top. You can never end up *less* secure than Mode 1.

### 5.1 Mode 1 — Sovereign (the base)

Zero PII. No server-side password. The only ways in are things the user possesses: an enrolled device, or a seed. This is the floor for every codex.

**The three doors (per D1):**

| Door | What it is | When used | Strength |
|---|---|---|---|
| **Passkey** | A WebAuthn credential bound to one device's hardware enclave, gated by biometric/PIN, non-exportable, origin-bound (phishing-resistant) | **Daily login** on an enrolled device | Strong; hardware-bound |
| **Standard key** (Seed-A) | The daily Apollo private key, typed from the seed | **Bootstrapping a brand-new device**, or login when no passkey is enrolled | Strong; as safe as where the seed is typed |
| **Smart key** (Seed-B) | The guardian Apollo private key | **Recovery** root; also login from a trusted device (warn: use only on a host you trust) | Strong; the cold root |

A user logging in day-to-day just taps their passkey (Windows Hello PIN, a security key, or their phone via QR — see device policy below). They only touch a seed to enroll a new device or to recover.

**What the user provides at signup:**
- The wizard's 3 primitives (which produce the two CodexID seeds)
- Enrollment of the first device's passkey (the first daily door)
- **No username. No password. No email. No phone.**

**Wraps of CK in the codex:**
- `wrap_seed` (derived from the Standard key / Seed-A — undeletable recovery root)
- `wrap_smart` (derived from the Smart key / Seed-B — second undeletable recovery root)
- `wrap_passkey[]` (one entry per enrolled device — the daily doors)

**Server stores per Mode 1 user:**
- `apollo_standard_pubkey`, `apollo_smart_pubkey` (the CodexID — these double as the lookup key; no username needed)
- The enrolled passkey credential IDs + their associated `wrap_passkey[]` envelope entries
- The codex blob (single concatenated format per v0.1-design.md §3.4)
- Timestamps
- **No password verifier, no PII columns.**

**Daily login flow (enrolled device):**
1. User opens the SPA; it offers "unlock with this device."
2. WebAuthn ceremony → biometric/PIN → PRF output → KEK → unwrap CK from this device's `wrap_passkey` entry.
3. SPA decrypts codex → user is in. (Server only ever sees the encrypted envelope; the unwrap happens locally.)

**Bootstrap a new device (enroll a passkey):**
1. On the new device, user chooses "add this device."
2. SPA requires **both** keys — Standard (Seed-A) and Smart (Seed-B) — the 2-of-2 enrollment gate (per D1). Typically done by approving from an already-enrolled device (QR + phone confirm) or by typing both seeds.
3. SPA unwraps CK (via wrap_seed/wrap_smart), runs a fresh WebAuthn registration on the new device, computes a new `wrap_passkey` entry, uploads it.
4. New device now has a daily door. Either seed alone could NOT have done this — that's the standing-backdoor protection.

**Recovery flow (lost all devices):**
1. User goes to recovery URL.
2. Enters **either** seed (Seed-A *or* Seed-B; any of the 4 input formats).
3. SPA derives the matching Apollo key → checks chain for the CodexID → fetches the encrypted envelope from server → unwraps CK from `wrap_seed` or `wrap_smart` → decrypts codex.
4. SPA prompts to enroll a fresh passkey on the current device. (Re-establishing a daily door is itself a device enrollment, so it asks for the *other* seed too — satisfying 2-of-2 — unless the user is content to keep logging in by seed.)
5. User is in.

**Device policy (per D1):**
- **Default offered: phone passkey** (Android/iOS secure element — least malware-prone).
- **For desktop browsers with no biometric:** the recommended path is **phone-via-QR** (WebAuthn cross-device "hybrid" — the desktop shows a QR, the phone's passkey signs). A **security key (YubiKey etc.) is also fully endorsed.**
- **Windows Hello / desktop-OS passkeys** are *soft-discouraged* (a one-line "desktop OSes are more exposed than phones" note) but **never hard-blocked** — power users can enroll them.
- Security keys are **never** discouraged.

**Custody property:** STRICT self-custody. Mnemosyne holds only encrypted envelopes and cannot decrypt anything. Lose every enrolled device AND both seeds → codex permanently inaccessible.

### 5.2 Mode 2 — +Password (= Mode 1 + a web2 convenience door)

Adds a username + password login door for users who want the familiar web2 experience. This is a **convenience layer, never a recovery root** — the strong Mode 1 doors remain fully present. It is the *weakest* door in the codex, which is the honest cost of the convenience.

**What the user provides at signup:** everything from Mode 1, plus a username (unique server-side, non-PII) and a password.

**Wraps of CK in the codex (additions to Mode 1):**
- `wrap_password` (CK wrapped with an Argon2id-derived key from the password)

**Server stores (additions to Mode 1):**
- `username` (lookup alias)
- `password_verifier` (Argon2id hash of password + salt), `salt_pw`, `salt_verifier`

**Login flow (Mode 2 door):**
1. User enters username + password.
2. Server returns the encrypted envelope + salts; verifier check passes.
3. SPA derives the password key (Argon2id, `salt_pw`) → unwraps CK from `wrap_password`.
4. SPA decrypts codex → user is in.

**Login flow (Mode 1 doors still work):** passkey / Standard key, exactly as Mode 1.

**Recovery flow:** same as Mode 1 (either seed). The password is *not* a recovery root — forget it and you reset it via a seed, never via the operator.

**Sensitivity note:** adding or changing the password layer needs only one factor (it's just a convenience door, §3.3). But because it's the weakest wrap, the wizard frames it honestly: "anyone who guesses this password gets in — keep it strong, and treat your seeds as the real keys."

**Custody property:** STRICT self-custody. The operator still cannot decrypt; the password wrap is computed and unwrapped client-side.

### 5.3 Mode 3 — +Email (= Mode 2 + email recovery path)

Adds an email-based recovery path. Email is a recovery convenience, **not a login door and not operator-held decryption material**.

**What the user provides at signup:** everything from Mode 2, plus an email address.

**Wraps of CK in the codex (additions to Mode 2):**
- `wrap_email_recovery` (CK wrapped with a key derived from the Email Recovery Secret)

**The Email Recovery Secret:**
- Random 256-bit value generated at signup.
- Emailed ONCE immediately, with strong "save this — we cannot resend" framing.
- Mnemosyne stores ONLY the wrap, never the secret.

**Server stores (additions to Mode 2):**
- `email` (lookup for the reset flow), `email_verified_at`.
(The `wrap_email_recovery` lives in the codex envelope, not as a separate server column.)

**Login flow:** same as Mode 1/2 doors. Email is for recovery, not login.

**Recovery flow (in addition to the seed paths):**
1. User clicks "Forgot access" → enters email.
2. Mnemosyne emails a magic link (1h validity).
3. User clicks link → server validates token → SPA prompts "paste your Email Recovery Secret."
4. User pastes secret → SPA derives the email-recovery key → unwraps CK from `wrap_email_recovery`.
5. SPA prompts to set a new password (if Mode 2 present) and/or enroll a fresh passkey.
6. User is in.

**Custody property:** hybrid. Recovery surfaces are the two seeds AND the user-held Email Recovery Secret. None involve operator-decryptable material.

### 5.4 Mode 4 — +Phone *(DEFERRED to v0.2)*

> **Not implemented in v0.1.** Per D2 and the pre-existing decision A4 ("Defer SMS to v0.2"), the phone/SMS tier is out of scope for the v0.1 launch. The design below is retained as the **intended v0.2 shape**, not a v0.1 deliverable. Running an SMS-sending architecture is operationally heavy and is the explicit reason for deferral.

Intended addition (v0.2): phone-based recovery and personal-facts recovery for grandma-tier accessibility — everything from Mode 3 plus a phone number (SMS-verified), 4–7 personal facts, and a user-chosen secret word.

Intended wraps (v0.2): `wrap_sms_recovery` (key from an SMS Recovery Secret) and `wrap_facts` (Argon2id-128M key from canonical facts + secret word). Phone would be a *recovery* surface only, never a login door (the SMS-only-auth rejection in v0.1-design.md §5.2 still stands).

**Grandma trade-off (carried forward to v0.2):** `wrap_facts` is the one wrap theoretically vulnerable to operator brute-force IF facts are weak and the operator has full DB access. Argon2id-128M makes this slow and costly but not cryptographically impossible — the explicit accessibility trade-off, to be re-confirmed when v0.2 is specced.

## 6. Mode is strictly cumulative

Each mode *adds* doors and recovery surfaces on top of the secure Mode 1 base — it never removes one. You can never end up less secure than Mode 1.

| Mode | Identifier(s) | Login doors | Recovery surfaces (additive) |
|---|---|---|---|
| 1 (Sovereign) | CodexID (pubkeys) | passkey (per device), Standard key | Seed-A, Seed-B |
| 2 (+Password) | CodexID, username | passkey, Standard key, **username+password** | Seed-A, Seed-B |
| 3 (+Email) | CodexID, username, email | passkey, Standard key, username+password | Seed-A, Seed-B, **email-secret** |
| 4 (+Phone) *(v0.2)* | CodexID, username, email, phone | (same — phone is recovery-only) | Seed-A, Seed-B, email-secret, **SMS-secret, facts** |

The passkey/seed doors of Mode 1 are present in every mode. Higher modes only add weaker, more convenient alternatives.

### Mode-switching

Users can change modes after signup. **Any mode transition is a sensitive operation** — it changes the codex's door/recovery surface — so it requires the **Standard + Smart 2-of-2 cosign** (§3.3).

**Adding a mode (e.g., Mode 1 → Mode 3):**
1. User logged in (SPA has CK in memory).
2. Wizard collects the new mode's credentials (password, or email + Email Recovery Secret generation, etc.).
3. SPA requires the Standard + Smart cosign (adding a door expands the trust surface).
4. SPA computes the new wrap(s) of CK and uploads them + the new server-side row data.
5. User now has both the Mode 1 doors AND the newly-added door.

**Removing a mode (e.g., Mode 3 → Mode 2):**
1. User logged in; Standard + Smart cosign required.
2. Wizard confirms: "this deletes the email-recovery path; you'll recover via your seeds only."
3. **24-hour cooling-off period** begins (mitigates account takeover via a single compromised door).
4. After cooling-off + explicit confirm, server deletes the higher-mode wraps + metadata.
5. Codex blob and CK untouched; the Mode 1 base wraps (`wrap_seed`, `wrap_smart`, `wrap_passkey[]`) are never removable.

**Invariants preserved across all mode transitions:**
- CK never changes
- Codex blob never re-encrypts (only the wrap set evolves)
- CodexID never changes
- CodexGuard never changes
- Duo Pure Prime never changes
- `wrap_seed` and `wrap_smart` are never removable (the two recovery roots are mode-independent)
- The on-chain `mnemosyne.CODEX` row never changes (mode-layer is purely off-chain)

## 7. Preset profiles

Pre-configured wizard sequences that minimize friction for specific user populations.

### 7.1 Migration preset (silent OuronetUI migration)

For existing OuronetUI users moving to Mnemosyne via the hard-cutoff migration. Designed to require zero user decisions beyond clicking "Migrate."

**Configuration:**
- **Onboarding path:** B (browser localStorage) or C (JSON backup), whichever the user has
- **Primitive 1:** autopilot → derive CodexID from existing Prime Codex Seed via Apollo curve
- **Primitive 2:** existing (no action — codex already has a CodexPrime)
- **Primitive 3:** autopilot → CodexGuard auto-generated in pure mode; existing Kadena seed positions continue to back CodexPrime
- **Mode:** Mode 1 (Sovereign) — the migrating user already owns a seed, so the secure base needs nothing new invented. A passkey is enrolled on the migrating device as the daily door. They can opt up to Mode 2/3 later.

**User experience:** click "Migrate to Mnemosyne" on devwallet.ouronetwork.io → unlock codex with existing password → enroll a passkey on this device → confirm. Done.

### 7.2 Movie Platform preset (frictionless grandma onboarding)

For users signing up on the movie platform who don't know what a codex is.

**Configuration:**
- **Onboarding path:** A (new codex from scratch)
- **Primitive 1:** autopilot → one split seed (24 Koala words, generated silently, NEVER shown — the one-split path of §3.1: smaller half → Standard, bigger → Smart). The "single hidden artifact" model is why this preset uses one split seed rather than the default two.
- **Primitive 2:** autopilot → reuse the seed via Dalos curve
- **Primitive 3:** autopilot → CodexGuard + Duo Pure Prime, all pure-mode
- **Mode (v0.1):** **Mode 3 (+Email)** — password + email recovery. The full phone/facts grandma flow (Mode 4) is **deferred to v0.2**; until then the most convenient available preset tops out at Mode 3.
- **Mode (v0.2 target):** Mode 4 (+Phone) — phone + password + facts, once SMS infra ships.

**User experience (v0.1):** "Sign up for movies" → email + password → done. Codex exists in the background; user has no awareness of it. (v0.2 adds the phone + personal-facts path.)

**Graduation path:** if the user later wants to graduate to the Sovereign base (Mode 1), the Mnemosyne SPA reveals the previously-hidden material (CodexID seed words, the Smart half, Duo Pure Prime keys, CodexGuard) with strong "save these now, you'll need them" framing, and prompts to enroll a passkey.

### 7.3 Optional minimal-friction variant of the Movie Platform preset

For users who want one touch of agency but otherwise full autopilot: offer a single optional step at signup — "Do you want to define your own Codex Identity seed words?" — defaulting to "no, just create one for me." If yes, the user types their own seed; everything else stays autopilot.

## 9. Recovery flows summary

All modes share the **universal recovery path**: provide **either** Codex Identity seed (Seed-A *or* Seed-B, in any of 4 input formats) → derive the matching Apollo key → unwrap CK from `wrap_seed` (Standard) or `wrap_smart` (Smart) → re-establish a daily door (enroll a fresh passkey, reset a password). This works even without Mnemosyne backend access if the user has the codex blob (from L1, L2 Arweave, or a JSON backup). Two independent recovery roots means losing one seed is not fatal.

Mode-specific recovery surfaces ADD to this universal path:

| Mode | Universal | Mode-specific |
|---|---|---|
| 1 (Sovereign) | either seed (Seed-A or Seed-B) | (none additional — passkeys are LOGIN doors, re-enrolled after a seed recovery) |
| 2 (+Password) | either seed | (none — password is a LOGIN door, reset via seed, never a recovery root) |
| 3 (+Email) | either seed | email magic link + Email Recovery Secret |
| 4 (+Phone) *(v0.2)* | either seed | SMS code + (SMS Recovery Secret OR personal facts + secret word) |

**Disaster recovery (Mnemosyne entirely gone):**
- If user has Codex Identity seed AND a copy of the encrypted blob (from Arweave L2 backup, JSON export, or any third party) → full recovery via standalone tool / any Mnemosyne-compatible client
- If user has only their seed but no blob copy → must rebuild codex from on-chain accounts (whatever they registered on Stoa is still there; address book + watchlist + Mode-specific settings lost)
- If user has only the blob but no seed → blob is opaque; no recovery
- If both lost → no recovery (same outcome as today's "lose seed = lose codex")

## 10. Open items pending owner refinement

Items to confirm before locking this document:

1. **Mixed Primitive 3 variants** — CodexGuard pure-mode + Duo Pure Prime from seed, or vice versa. Currently spec'd as supported via Advanced tab. Confirm or simplify.

2. **Movie Platform Smart-half handling.** For grandma users who never see their seed (the one-split autopilot path, §3.1; Mode 3 in v0.1, Mode 4 in v0.2):
   - (a) Generate the Smart half silently, store inside codex, never reveal unless user graduates → Smart-cosign + second-recovery-root defense available but invisible (**recommended** — keeps the 2-of-2 enrollment gate meaningful)
   - (b) Surface the Smart half to the user as her only visible artifact → user must save it, breaks the "no friction" promise
   - (c) Skip Smart-half generation entirely → no cosign defense, but consistent with the "she'll never need it" framing. **Note:** under D1 this also removes the second recovery root, so it should be weighed carefully.

3. **24-hour cooling-off duration** for mode removal — tunable. Confirm or adjust.

4. **Standard-key-compromise recovery** — current design has NO rotation path; only recourse is migrate to a fresh CodexID. Acceptable for v0.1, or include a Standard-key-rotation flow at consumer-app level (e.g., consumer re-issues identity after compromise notification)?

5. **Off-chain Schnorr verification protocol** — what exact format do consumers use to issue challenges and submit responses? Possible answers: OAuth-pattern with custom Apollo-sig grant type / simple HTTP challenge-response / deep-link relay between consumer and Mnemosyne SPA. Deserves its own spec doc once the rest of v0.1 is locked.

6. **Smart Apollo glyph prefix** — currently `N.` as a placeholder in the CodexID address format `₱.STANDARD.N.SMART`. Owner: pick the actual glyph (`Ñ.`, `Π.`, `Σ.`, custom?).

7. **Server-side row format for CodexID** — single string `₱.STANDARD.N.SMART` as a PK, or two columns `apollo_standard_pubkey` + `apollo_smart_pubkey` with a separate UUID PK? Affects chain row schema in `mnemosyne.CODEX` too.

8. **Migration preset behavior on chain registration collision.** If a migrating user's existing Prime Codex Seed somehow derives a CodexID that's already registered on chain (vanishingly improbable but worth specifying), how does the wizard handle it? 9. **Per-mode wizard step ordering** — current spec lists steps but doesn't lock the exact UI sequence. Worth a wireframe pass once the spec stabilizes.

10. **The Apollo curve operations themselves** — derivation, signing, verification. The dalos-crypto package already implements these (per the example shown in design discussion). Confirm the package's current surface covers everything this doc needs, or note gaps.
