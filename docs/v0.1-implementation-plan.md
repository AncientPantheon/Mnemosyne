## Phase 2 — Mnemosyne backend MVP

**Duration estimate:** 2-3 weeks
**Owner:** Epyc
**Depends on:** Phase 0 (decisions) + Phase 1 (v0.3.0 codex)
**Blocks:** Phase 3, Phase 5

### Deliverables

#### 2.1 Database schema

SQLite via better-sqlite3 (assuming A2 = SQLite). Tables:

```sql
CREATE TABLE users (
  id TEXT PRIMARY KEY,                -- UUID
  primary_identifier_type TEXT NOT NULL,  -- 'phone' | 'email' | 'username' | 'wallet'
  primary_identifier_value TEXT NOT NULL,  -- normalized form
  password_verifier TEXT NOT NULL,     -- Argon2id(password, salt)
  codex_prime_address TEXT,            -- nullable; populated after codex kickstart
  herotag_name TEXT,                   -- nullable; cached from on-chain
  recovery_facts_hash TEXT,            -- nullable; only for grandma-tier users
  created_at TEXT NOT NULL,            -- ISO 8601
  last_login_at TEXT,
  UNIQUE (primary_identifier_type, primary_identifier_value)
);

CREATE TABLE user_identity_modes (
  user_id TEXT NOT NULL,
  mode_type TEXT NOT NULL,             -- 'phone' | 'email' | 'username' | 'wallet'
  mode_value TEXT NOT NULL,
  added_at TEXT NOT NULL,
  PRIMARY KEY (user_id, mode_type),
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE codex_blobs (
  user_id TEXT PRIMARY KEY,
  encrypted_codex BLOB NOT NULL,                -- AES-256-GCM ciphertext
  encrypted_codex_key_password BLOB NOT NULL,   -- CK wrapped by password-derived key
  encrypted_codex_key_facts BLOB,               -- nullable; only for grandma-tier
  schema_version INTEGER NOT NULL,              -- codex package's data schema version
  version INTEGER NOT NULL,                     -- monotonic; for optimistic concurrency
  last_updated_at TEXT NOT NULL,
  last_updated_by_device TEXT,                  -- 'dev' | 'main' | client-supplied label
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  device_label TEXT,
  ip TEXT,
  user_agent TEXT,
  created_at TEXT NOT NULL,
  expires_at TEXT NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE partner_apps (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  app_key_hash TEXT NOT NULL,
  gas_budget_address TEXT,
  created_at TEXT NOT NULL
);

CREATE TABLE audit_log (
  id TEXT PRIMARY KEY,
  user_id TEXT,                                -- nullable for unauthenticated events
  event_type TEXT NOT NULL,                    -- 'signup', 'login', 'password_reset', 'codex_update', etc.
  ip TEXT,
  user_agent TEXT,
  detail TEXT,                                 -- JSON for arbitrary structured data
  occurred_at TEXT NOT NULL
);

CREATE TABLE pending_password_resets (
  token TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  reset_method TEXT NOT NULL,                  -- 'mnemonic' | 'facts' | 'email-link'
  expires_at TEXT NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

Migrations live in `db/migrations/` like AH; first migration creates the schema.

#### 2.2 Auth surface

- `POST /api/auth/signup` — accepts `{ mode, identifier, password, ... }`; runs Argon2id; creates user row; returns session JWT
- `POST /api/auth/login` — verifies password; returns session JWT
- `POST /api/auth/logout` — invalidates session
- `POST /api/auth/add-mode` — adds a secondary identity mode (sends verification code for phone/email)
- `POST /api/auth/remove-mode` — removes a mode with 24h cooling-off (writes to pending table)
- `POST /api/auth/verify-mode` — verifies the verification code

Argon2id parameters tuned for ~500ms per hash; salt per-user; constants in `lib/auth-config.ts`.

#### 2.3 Codex CRUD surface

- `GET /api/codex` — returns `{ encrypted_codex, encrypted_codex_key_password, version, schema_version }`
- `PUT /api/codex` — accepts `{ encrypted_codex, encrypted_codex_key_password, version }` with `If-Match: <prev_version>` header; returns 409 on conflict
- `POST /api/codex/change-password` — accepts `{ old_encrypted_key, new_encrypted_key }`; verifies old password; updates wrapper
- `POST /api/codex/reset-password` — initiates recovery (mnemonic-based; for grandma-tier the facts-based path)
- `GET /api/codex/download` — returns the V1.2 backup JSON for self-custody export

#### 2.4 Herotag proxy surface (Mnemosyne caches on-chain reads)

- `GET /api/herotag/resolve?name=<name>` — proxies chain read; caches for 30s
- `GET /api/herotag/availability?name=<name>` — checks chain `is-available`; caches
- `POST /api/herotag/claim` — prepares the Pact tx for the user to sign client-side; returns the unsigned tx + metadata
- `POST /api/herotag/confirm-claim` — user-signed tx received; submits to Stoa via the gas station; updates user.herotag_name cache

#### 2.5 Partner-app onboarding surface

- `POST /api/onboard` — partner app calls with `{ partner_app_key, identifier, password, tier, facts? }`; mints codex, returns ouro address

#### 2.6 User management surface

- `GET /api/user` — current user account info
- `DELETE /api/user` — wipes everything (irreversible; confirmation token required)

#### 2.7 Cryptography helpers

- `lib/crypto.ts` — pure functions: derive password verifier, derive password-wrap-key, AES-256-GCM wrap/unwrap, generate random CK
- All cryptographic operations happen here; rest of backend treats it as a black box
- Client-side operations (in Phase 3) re-implement the SAME logic in JS

#### 2.8 Tests

- Vitest unit tests for `lib/crypto.ts` (~10 tests)
- Vitest integration tests for each API endpoint (~30 tests)
- Use a fresh in-memory SQLite per test for isolation

#### 2.9 Deployment

- Hetzner VPS (matches AH operations pattern)
- nginx reverse proxy: `api.mnemosyne.ancientholdings.eu` → Node process on `:3001`
- PM2 to manage the Node process
- Migrations run automatically on boot
- Boot logs banner the current commit SHA + version

### Acceptance

- All endpoints respond per the API spec
- A user can sign up via curl, get a JWT, store an encrypted codex, retrieve it, change password, delete account
- Audit log populates correctly
- Pre-flight script verifies: env vars set, DB writable, migrations applied
- Production deploys cleanly to Hetzner VPS

### Out of scope for Phase 2 (deferred)

- 2FA (TOTP) — v0.2
- Rate limiting infrastructure — v0.2 if abuse emerges
- Arweave snapshots — v0.2
- On-chain Stoa pointers — v1.0

**Duration estimate:** 1 week

6. ✅ Documentation of the deployment process — keyset, governance, addresses

### Critical detail — immutable governance setup

The module's governance keyset MUST be set up such that NO ONE can ever upgrade the module. Two acceptable patterns:

**Pattern A — Unsatisfiable keyset:**
```pact
(define-keyset 'mnemosyne-herotag-governance
  { 'keys: ["MNEMOSYNE_GOVERNANCE_FROZEN_FOREVER_KEY_DO_NOT_USE"]  ; not a valid public key
  , 'pred: keys-all })
```
No private key exists that satisfies this. The "key" string is intentionally invalid.

**Pattern B — `enforce false` in defcap:**
```pact
(module mnemosyne-herotag GOV
  (defcap GOV () (enforce false "module is immutable by design"))
  ...)
```
Module's GOV capability ALWAYS fails. No upgrade path can satisfy it.

**Recommend Pattern B** — it's cryptographically guaranteed unsatisfiable (no edge case where someone produces the magic string). Pattern A relies on the assumption nobody breaks SHA-256 to forge a key matching the string hash. Pattern B is true forever.

### Acceptance

- Module deployed to StoaChain testnet
- Test claim succeeds; test resolution returns correct address
- Test duplicate-claim fails atomically
- Test unauthorized-claim (wrong signer) fails with capability violation
- Documentation explains how to verify the module is immutable (publish the governance setup proof)

### Out of scope

- Reverse-resolve (address → herotag) — adds another index; not needed for v0.1
- Bulk operations (claim N herotags in one tx) — gas cost not worth it
- Transfer / reassign — by design impossible (immutability)

## Phase 9+ — Decentralization layers

### Phase 9 (v0.2): Arweave vault snapshots

- Mnemosyne server batches a snapshot of the entire encrypted vault state weekly (per user's preference articulated 2026-05-27)
- Posts to Arweave (~$1/week for the entire user base at current pricing)
- Disaster-recovery: anyone with the Arweave tx-id can fetch the encrypted blob; user can decrypt with their password
- ~1 week of work

### Phase 10 (v0.3): Per-codex Arweave pinning

- User opts in (paid tier) to have THEIR specific codex blob pinned individually
- More expensive than vault snapshots ($0.50-1 per codex update) but gives per-codex permanence guarantees
- ~1 week of work

### Phase 11 (v1.0): On-chain Stoa pointers

- Pact module `mnemosyne-vault` deployed on Stoa
- Each user's codex pointer (`{ ouro-addr → arweave-tx-id of latest blob }`) lives on-chain
- Ownership cryptographically enforceable; third-party Mnemosyne-compatible clients can interop
- ~2-3 weeks of work; significant migration testing

## 15. What this plan does NOT cover

Out of scope by design — these need separate plans:

- **Mnemosyne mobile apps** — iOS/Android native clients. The SPA works on mobile browsers; native apps are a v0.5+ topic.
- **Browser extension** — Mnemosyne as a MetaMask-equivalent extension. Possible but not v0.1.
- **Hardware wallet integration** — Ledger / Trezor for the prime seed. Future capability.
- **Multi-sig codex** — codex owned by M-of-N humans. Genuinely interesting; v1.5+ research.
- **Codex sharing / family vaults** — multiple users having read-only access to one codex (e.g., couple's joint wallet). Not v0.1.
- **The actual movie platform** — separate project.
- **Branding / marketing / landing-page copy** — content + design work, separate from the engineering plan above.
