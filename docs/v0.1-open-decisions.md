# Mnemosyne v0.1 — Open decisions

**Status:** decisions list (2026-05-27)
**Companion to:** [`v0.1-design.md`](./v0.1-design.md) (the design) and [`v0.1-implementation-plan.md`](./v0.1-implementation-plan.md) (the build plan)

Each decision below has my recommendation and the trade-offs. Lock these in order of criticality — the "Must-decide-before-Phase-1" group blocks all subsequent work.

---

## Group A — Must decide BEFORE Phase 1 starts

These are upstream — they shape what gets built. Locking these unblocks everything.

### A1. Backend framework

**Options:**
- **Next.js** (matches AH hub stack; full-stack React-friendly; SSR if needed)
- **Hono** (lightweight, modern, edge-runtime-capable, no React baggage on backend)
- **Express** (oldest, most stable, biggest ecosystem)
- **Fastify** (faster than Express, plugin-based)

**My recommendation:** **Next.js**. We already operate one (AH hub), team knows it, deploy flow is identical to AH's. Backend lives in API routes; if we later want the SPA + backend to be one Next.js project (a single deployment instead of two), the option is open. Hono would be cleaner for pure-API, but the operational simplicity of "another Next.js box" wins.

**Locking this decides:** Phase 2 starts immediately.

### A2. Database engine

**Options:**
- **SQLite via better-sqlite3** (matches AH; zero ops; single-server)
- **PostgreSQL** (scales horizontally if we grow; ops overhead)
- **Turso** (libSQL, edge-distributed SQLite; managed)

**My recommendation:** **SQLite via better-sqlite3** to start. Same pattern as AH hub. If/when Mnemosyne hits ~50k+ active users we'd reconsider Postgres, but that's far off and the migration path is well-trodden. Single server runs fine for the entire StoaOuronet user base for years.

**Locking this decides:** Phase 2 backend schema work + Phase 7 (AH integration uses same engine).

### A3. Auth library

**Options:**
- **Roll our own** (Argon2id + JWT, ~1 week build): full control, fewer deps, exact behavior we want
- **Lucia** (lightweight, modern, opinionated about sessions)
- **Auth.js / NextAuth** (Next.js-native, lots of providers built-in, opinionated)
- **Supabase** (fastest to ship, vendor lock-in, hosted PostgreSQL too)

**My recommendation:** **Roll our own** — the auth surface is unusual (multi-mode: phone/email/wallet/username), the password-wrapping crypto envelope is custom (per §3.3 of design doc), and we'd end up writing the custom parts anyway. Off-the-shelf libraries add abstraction without value here. Cost: ~1 week of focused work, including Argon2id tuning + OPAQUE-style flow + session JWTs.

**Locking this decides:** Phase 2 auth scope. If user picks Lucia/Auth.js, that's another evaluation week before Phase 2 can start.

### A4. SMS provider (only if Streamlined Mode ships in v0.1)

**Options:**
- **Twilio** (gold standard; ~$0.04/SMS; reliable everywhere)
- **MessageBird** (cheaper for EU; ~$0.03/SMS)
- **AWS SNS** (cheapest at scale; more setup)
- **Defer SMS entirely** — ship Streamlined Mode in v0.2; only Privacy + Wallet + Convenience (email) modes at v0.1

**My recommendation:** **Defer SMS to v0.2**. Streamlined Mode (phone) was justified by the grandma-onboarding-via-movie-platform use case, but the movie platform isn't built yet. Shipping email-only (Mode 3 Convenience) covers OuronetUI migration + immediate value. SMS adds Twilio operations + costs + a new attack vector — defer until grandma actually exists.

**Locking this decides:** v0.1 scope. If user wants SMS now, add ~1 week.

### A5. Hosting

**Options:**
- **Hetzner VPS** (matches AH; ~€10/month per server; full control; you own the box)
- **Managed (Vercel + PlanetScale/Supabase)** (easier ops; vendor lock-in; more expensive at scale)

**My recommendation:** **Hetzner VPS**. Same as AH; you already have the operational pattern; SQLite means no separate DB service needed. Spin up one new VPS for `api.mnemosyne.ancientholdings.eu` + serve the SPA from `mnemosyne.ancientholdings.eu` (same nginx, different vhost — could even be the same box at low scale).

**Locking this decides:** Phase 2 deploy story + Phase 3 SPA host.

### A6. Where do Mnemosyne credentials get registered (env / vault)

**Options:**
- **`.env.local` on the VPS** (simplest; AH pattern)
- **Vault tier (lib/vault.ts equivalent)** (encrypts at rest with master key)
- **External secrets service (Doppler, Infisical)**

**My recommendation:** **`.env.local`** for v0.1. Mnemosyne has fewer secrets than AH (no SSH keys, no per-user data — just the server-side master key for session JWTs + the SMS/email provider creds). Matches AH operational pattern.

### A7. Codex package v0.3.0 scope confirmation

**Options:**
- **Per the design doc §9.1**: Guardian + activation status + consumer settings registry + schema migration framework + IHerotag field (5 things, ~1-2 weeks)
- **Trim**: ship just Guardian + IHerotag now (~3-5 days); defer the rest to v0.3.x
- **Expand**: also add UiSettings cleanup in v0.3.0 (not v0.3.1)

**My recommendation:** **Per design doc § 9.1 as-is.** All 5 items are Mnemosyne-blockers; deferring any means an extra package cycle later. UiSettings cleanup IS deferrable (Mnemosyne uses consumerSettings directly, doesn't need OuronetUI's legacy keys moved).

**Locking this decides:** Phase 1 scope.

## Group B — Decide DURING Phase 1-2 (can defer up to ~2 weeks)

### B1. Multi-device sync conflict resolution

**Options:**
- **Optimistic concurrency** (version counter; 409 on conflict; client retries)
- **Last-writer-wins** (simplest; can silently overwrite)
- **CRDT-based merge** (complex; auto-merges; overkill)

**My recommendation:** **Optimistic concurrency**. UX: if both devices write within the same second, the second one gets "your codex was updated elsewhere — please reload" prompt. Rare in practice (users don't mutate codices that often).

### B2. Session token strategy

**Options:**
- **JWT** (stateless; no DB lookup per request; harder to revoke)
- **DB-backed sessions** (revocable; one DB lookup per request)

**My recommendation:** **DB-backed sessions** for v0.1. Revocability matters for crypto (compromised session = drained codex if attacker has the active session token). The per-request DB lookup is fine at SQLite-speed; we're not Twitter.

### B3. API rate limiting strategy

**Options:**
- Per-IP rate limits
- Per-user rate limits
- Both

**My recommendation:** **Both, with stricter per-user on sensitive endpoints** (password reset, recovery facts validation). Standard pattern.

### B4. DB encryption at rest

**Options:**
- **SQLCipher** (SQLite with built-in AES encryption)
- **OS-level disk encryption only**
- **None** (relying on access controls)

**My recommendation:** **SQLCipher**. The codex blobs in the DB are encrypted at the application layer anyway (server can't read them), but SQLCipher adds defense-in-depth for the metadata layer (which emails exist, when accounts were created, etc.).

### B5. Email provider for Convenience-Mode verification

**Options:**
- **Postmark** (best deliverability for transactional; ~$10/month)
- **SendGrid** (popular; can drift into marketing-email category)
- **AWS SES** (cheapest; requires reputation management)
- **Use existing AH SMTP credentials** (the contact form in AH already sends via nodemailer; could reuse)

**My recommendation:** **Reuse AH SMTP credentials** for v0.1, switch to Postmark if delivery issues emerge. Low setup cost.

### B6. Herotag namespace policy

**Options:**
- **First-come-first-served, no reservation** (pure permissionless)
- **Reserved names** (`@admin`, `@support`, `@mnemosyne`, `@ouronetwork`, etc.)
- **Verified-badge tier** (paid verification for known entities)

**My recommendation:** **Reserve ~20 obvious admin/system names**; first-come-first-served otherwise. Verified-badge tier is a v0.2+ revenue model (Twitter blue-check pattern); not needed at launch.

## Group C — Defer to Phase 9+ (post-launch)

### C1. Pricing model

**Options:**
- Free for all users (eat the gas-station cost)
- Per-codex Arweave-pinning paid tier ($1-5/month)
- Bundle subscription with movie platform / OuronetUI Pro tier

**My recommendation:** **Free for v0.1**. The gas-station cost per onboarding is ~$0.001 (activation tx); Mnemosyne itself has marginal hosting cost. Revenue model is a v0.2+ business decision (likely the Arweave-pinning tier).

### C2. Self-hosting story

Is Mnemosyne open-source enough that someone could run their own?

**My recommendation:** **Yes, open-source under MIT or Apache-2.0** — this aligns with the "jurisdictionless" vision the user articulated. Anyone can run their own Mnemosyne instance pointing at any Stoa-chain (testnet, mainnet, sidechain). Doesn't change v0.1 build cost; just license + repo visibility.

### C3. Geographic deployment

**My recommendation:** **EU-only at v0.1** (Ancient Holdings GmbH is EU-based; lower regulatory complexity to start). Add US / global CDN at v0.2 if traction warrants.

### C4. Mnemosyne in cross-pollinate workspace

**My recommendation:** **Register after Phase 2 (backend MVP) lands**. The cross-pollinate config tracks publishers — Mnemosyne becomes one once it has its own publishable npm package (`@stoachain/mnemosyne-client` in Phase 5). Until then, it's just a sibling dir.

### C5. OuronetUI migration sequencing

**My recommendation:** Per design doc §5.5 — **hard cutoff ~6 months after Mnemosyne v0.1 ships**. Until cutoff, OuronetUI v1.4.0+ offers "Sync with Mnemosyne" as opt-in; cutoff date publishes the mandatory migration.

### C6. AH hub integration with Mnemosyne

**Question:** Does the AH hub's own codex (currently SECRETS_MASTER_KEY-encrypted server-side) migrate to Mnemosyne?

**My recommendation:** **No** for v0.1. AH hub is a single-codex, autonomous-signing system; Mnemosyne is multi-user. The hub's codex doesn't need cross-device sync. BUT — AH hub could use Mnemosyne for STORAGE redundancy (Mnemosyne holds the encrypted blob; AH hub still has the master key locally to unwrap). Useful in v1.0+.

### C7. SureMark integration

**My recommendation:** **Watch + maintain Stuart Haber relationship**, but no integration work at v0.1. Revisit when Mnemosyne has herotag adoption + when SureMark has a stable consumer-facing API.

### C8. Movie platform integration spec

**Recommendation:** **Wait for movie platform tech stack** to be picked. Then write the partner-app onboarding API to match what they need.

### C9. Hard-cutoff date for OuronetUI migration

**Recommendation:** Pick a concrete date once Mnemosyne v0.1 is stable + has been live for ~3 months. Probably **2027 Q1**.

## Group D — Identity & login model refinements (raised 2026-05-30)

Raised in the 2026-05-30 design session after colleague feedback that the current Mode 1 (username + password) is a security regression for a Web3 codex: a server-side password is a full parallel door to CK, so it negates the seed's strength, and each higher mode adds *more* weak doors to the same vault. These decisions **amend the canonical modes doc** ([`v0.1-onboarding-and-modes.md`](./v0.1-onboarding-and-modes.md) §5.1, §6) once locked.

### D1. Mode 1 login model — key possession, not username+password  — ✅ LOCKED (2026-05-30)

**The reframe:** authentication is proof of *key possession*, never knowledge of a server-side password. The chain is only as strong as its weakest wrap of CK, so Mode 1 must have **no weak door** — only strong, key-bound ones. Zero PII throughout.

**The three-tier login mechanism:**
- **Daily (enrolled device):** a **device passkey** (WebAuthn PRF) holds a key in the secure enclave that unwraps CK via a new `wrap_passkey`. Nothing is typed; hardware-bound, biometric-gated, phishing-resistant (origin-bound), and **revocable per device** (one `wrap_passkey` entry per device in the blob; lose a device → delete that entry, seed still recovers everything).
- **New-device bootstrap / untrusted machine:** type the **Standard private key alone** (not the full seed — keeps the Smart guardian key out of the browser) → unwrap CK via `wrap_seed` → enroll a passkey on this device → never type again here.
- **Recovery (lost everything):** type the **full seed** on a trusted device.

**Out of scope (explicitly, because impossible):** a browser "secure keyboard / blackbox" that defeats keyloggers, malicious extensions, or screen capture. A web page is the least-privileged process on the machine; there is no web equivalent of Android `FLAG_SECURE`. The only real defense against a compromised host is *not typing the secret* (passkey/enclave) — which is what this model does. Typing a key into an untrusted machine carries irreducible risk (same as importing a seed into a fresh MetaMask); we make that the *rare* event, not the daily one. Field masking (password dots) is retained — it addresses the camera/shoulder-surfer threat only, not malware.

**My recommendation:** adopt as above. Passkey daily / Standard-key bootstrap / seed recovery.

**Refinements (2026-05-30 cont.):**

- **Tiered access, mapping onto §3.3:**
  - **Unlock** (read/use in a session): any enrolled passkey **or** Standard key **or** Smart key.
  - **Enroll a new device / sensitive ops:** **both** Standard + Smart (2-of-2 cosign). Adding a device expands the trust surface → sensitive.
  - **Recover:** either key alone.
- **Smart key becomes a second unlock/recovery root** (a second wrap of CK keyed off Smart, alongside `wrap_seed` keyed off Standard). This removes the dangerous asymmetry where losing Standard = total loss — now losing *either* seed is survivable. Smart stays cold in normal life (only touched to enroll a device or recover); UI warns it's the high-value guardian key, use only from a trusted device.
- **Honest limit (must be documented, not oversold):** both-keys-to-enroll stops *persistent* backdooring (a single stolen key can't plant a standing passkey), **not** *in-session* draining (one key unlocks the blob, which holds the account keys). Stopping in-session theft is the separate on-chain CodexGuard / cosign-for-high-value-tx lever.
- **Device-type policy:** allow all device types, but **default to phone**; present **"sign in with your phone via QR"** as *the* recommended path for desktops (the phone keeps the key, the desktop only borrows it per-login — nothing standing on the weaker machine). **Soft-discourage with a warning** the specific case of a *platform passkey stored in a desktop OS* (esp. Windows — more malware-prone than iOS/Android). **Never discourage hardware security keys** (YubiKey-class) — those are *more* secure than a phone. Don't hard-block desktops (no-smartphone users).

**Locking this decides:** A3 auth scope, the Primitive-1 spec, and the §5.1 rewrite.

**LOCKED (2026-05-30):** adopted exactly as above — passkey daily / Standard-key bootstrap / either-seed recovery; Smart as second recovery root; 2-of-2 cosign to enroll a device or do sensitive ops; honest in-session-drain limit documented; device policy = default phone, phone-QR for desktops, soft-discourage desktop-OS passkeys, never discourage security keys, don't hard-block. The §5.1 / §3 rewrite in [`v0.1-onboarding-and-modes.md`](./v0.1-onboarding-and-modes.md) reflects this.

### D2. Password/username demoted to Mode 2+ (ladder inversion)  — ✅ LOCKED (2026-05-30)

Username + password leaves Mode 1 entirely and becomes an **opt-in convenience door added at Mode 2**. This makes the ladder honest: Mode 1 = maximally secure & private (one strong key-door); each step up *explicitly* trades security for recoverability/convenience. Note: this cannibalizes the old Mode 2 ("wallet-signature login"), since Mode 1 is now itself signature-based — old wallet-sign folds into Mode 1/2 as an optional factor, not its own tier.

**My recommendation:** adopt the inversion; redefine the 4-tier ladder in a follow-up modes-doc rewrite.

**LOCKED ladder (2026-05-30):**

| Mode | Adds over prior | Who it's for | Security direction |
|---|---|---|---|
| **1 — Sovereign** | two seeds + passkey/device-list (per D1) | crypto-natives, max privacy | strongest, zero PII |
| **2 — +Password** | username + password | web2 familiarity / consumer onboarding | first *weak* door, opt-in |
| **3 — +Email** | email reset path | recover without your seeds | weaker recovery door |
| **4 — +Phone/SMS** *(DEFER → v0.2, per A4)* | phone number + SMS + facts | movie-platform onboarding | weakest; build only when needed |

Key rulings behind this shape:
- **Passkeys stay in Mode 1, never gated to a higher mode.** Gating them re-inverts the security ladder (base would be weakest-in-daily-use again) and breaks the ladder's single meaning ("each step up trades security for convenience"). Passkeys are the safe *default* way to use a key, not a premium feature.
- **Password (Mode 2) is the web2 bridge, not a capability gain.** With passkeys in Mode 1, a password adds ~no convenience and is the weakest door — it earns its place only as the familiar credential mainstream/consumer-app users (e.g. movie platform `{email,password}`) expect. Label it as the weakest link; mitigate with rate-limiting + breach-notification; treat weak doors as recovery-with-delay/notification rather than silent full access.
- **SMS ≠ phone-passkey (not redundant, but Mode 4 still defers).** Phone passkey = strong hardware *unlock* bound to the device; SMS = weak number-based *recovery* (SIM-swap/SS7), bound to a transferable SIM. They don't overlap in security. SMS's onboarding-identity role is largely coverable by passkey + email, and running an SMS stack adds cost + ops + a new attack vector (see A4). **Ship v0.1 as Modes 1–3; defer Mode 4 to v0.2**, revisit only when the movie platform genuinely needs phone-number onboarding.

### D3. CodexID seed structure — one split seed vs two independent seeds

Today: one seed, system-split (smaller half → Standard, bigger → Smart, §3.6). Standard-only login is awkward under this (either type the whole phrase and expose Smart, or make the user find the split point). **Proposal:** let the user maintain **two independent seed sets** (Seed-A = Standard/daily, Seed-B = Smart/guardian), so Standard-only login is unambiguous ("type Seed-A"), the two can be stored separately (breach the convenient one → daily access only, not guardian-gated ops), and the mental model matches the roles.

**Trade-off:** two backups with asymmetric criticality — lose **Seed-A** in Mode 1 = total loss (no other recovery door); lose **Seed-B** = survivable (recover via Seed-A) but lose sensitive-op cosign permanently → forced migration to a fresh CodexID. Smart is still exposed at the moment of an actual cosign (whether Smart deserves its own guardian-device passkey is a downstream question).

**DECIDED (2026-05-30): two independent seeds** (Seed-A = Standard/daily, Seed-B = Smart/guardian) — chosen for the clearer user mental model (two distinct roles vs. one phrase the system secretly splits). One-split remains available as the autopilot/grandma-preset default where the seed is never shown to the user. Architecture is identical either way (D1 envelope unaffected); this is purely the input/backup UX.

**Locking this decides:** Primitive-1 wizard spec + CodexID derivation.

### D4. Cold-signer / air-gapped hardware tier — FUTURE R&D (post-v0.1)

**Out of scope for v0.1** (we are browser-only for now). Captured as a future research/development path. The need: a cold-storage device that holds the seed/keys and *emits signatures* (movable across the air gap via QR), so even a fully-compromised online machine never sees private key material.

**Findings from the 2026-05-30 discussion (for when we pick this up):**
- **Don't build a custom Linux distro.** Maintenance + supply-chain trust (users must trust our ISO) + arbitrary-hardware boot support make it a net liability. A homemade distro is *less* trustworthy than an audited one.
- **Software can't *guarantee* offline** — it can only *disable* networking (e.g., Tails offline mode). The real guarantee is **physical air-gap** (hardware with no radios). Gold standard = air-gapped hardware + amnesic offline OS (Tails).
- **Existing hardware wallets:** viable only on open firmware (Trezor/Keystone/BitBox; **not** Ledger, which gatekeeps + lacks our curves natively). The dealbreaker is our **custom Apollo/Dalos curves** — they'd need a secure embedded re-implementation (constant-time, side-channel-safe) on a constrained MCU, discarding the dalos-crypto code we already have, plus a firmware fork to maintain forever.
- **Recommended sweet spot:** a **Raspberry Pi** (radio-less model, e.g. Pi Zero non-W) or stripped old laptop running our **existing dalos-crypto signer as-is**, with a small screen for **what-you-see-is-what-you-sign**, QR in / QR out. ~5% of the firmware-fork effort, ~95% of the security. Custom-firmware hardware wallet (fork an open QR-native device like Keystone) is a v2+ track if whale/institutional demand appears.
- **Architecture fit:** drops into design-doc §3.4 unchanged — SPA constructs canonical payload bytes (online), the cold signer signs them (offline), signatures return online for submission. No core change.

**Related — Mnemosyne mobile app (also deferred):** the future app = the whole Mnemosyne **web experience wrapped in a mobile shell**, which also becomes the natural home for the on-device authenticator/signer (the "custom phone app" flavor of the phone-QR login) and the soft cold-signer. Same future bucket as the hardware track. For v0.1, phone login uses **standard OS passkeys** (phone-QR is built into the OS/browser — no app to build).

**My recommendation:** defer to post-v0.1; revisit as a hardware + mobile-app R&D track. No v0.1 build impact.
